## 1. 项目概述

### 1.1 目标
实现一个基于Kubernetes Sidecar模式的智能体浏览器自动化工具，支持通过直接函数调用（非MCP协议）被OpenClaw调用，完成浏览器自动化测试、登录验证和页面交互。

### 1.2 核心改变
- ✅ **放弃MCP协议**：改用直接本地函数调用（Unix socket/TCP）
- ✅ **放弃Browser Pool**：采用Sidecar模式，每个Pod独立Browser实例
- ✅ **放弃Redis会话存储**：改用共享Volume的加密文件存储
- ✅ **简化架构**：减少服务数量从5+到2容器/Pod
- ✅ **增强安全性**：减少攻击面，移除已知CVE风险

### 1.3 系统定位
```
OpenClaw调用方
        │
        │ (直接TCP/Unix Socket调用)
        ▼
┌─────────────────────┐
│   Sidecar Pod       │
│  ┌─────────────┐    │
│  │ MCP Server  │◄──►│ Browser Container │
│  │ (Node.js)   │    │ (Chrome + PW)     │
│  └─────────────┘    └─────────────────┘
│          │                   │
│          └───────┬───────────┘
│                  │
│          共享存储卷   │
│    (会话持久化加密文件) │
└─────────────────────────────────────┘
```

## 2. 架构决策

### 2.1 核心架构选择：Sidecar模式

**为什么选择Sidecar：**
- ✅ **故障隔离**：浏览器崩溃只影响单个Pod
- ✅ **资源精细控制**：可分别设置MCP Server和Browser的资源限制
- ✅ **低延迟通信**：同一Pod内通过localhost通信
- ✅ **会话共享**：共享Volume提供快速一致性存储
- ✅ **标准Kubernetes**：无需特殊运维知识
- ✅ **安全加固**：减少网络暴露点，移除MCP协议风险

### 2.2 通信方式：直接函数调用

**替代MCP协议的选择：**
- ❌ MCP协议：JSON-RPC over HTTP/STDIO（安全风险+性能开销）
- ✅ 本地TCP/Unix Socket：直接函数调用式通信
  - 零序列化开销（可使用Protobuf或简单JSON）
  - 无额外进程隔离
  - 原生错误传播
  - 简单易调试

### 2.3 会话存储：加密文件系统

**替代Redis的选择：**
- ❌ Redis：网络延迟+一致性风险+大value问题
- ✅ 加密文件：共享Volume存储
  - 本地文件系统速度（毫秒级）
  - 天然一致性（无网络分区问题）
  - 简单备份和迁移
  - 易于审计和合规

### 2.4 浏览器运行：独立Chrome实例

**替代Browser Pool的选择：**
- ❌ Browser Pool：连接泄露/Zombie进程/单点故障
- ✅ 每Pod独立Chrome：故障隔离+资源透明
  - 每个Pod拥有自己的Chrome实例
  - 浏览器崩溃只影响单个Pod
  - 资源使用精细可控（cgroup限制）
  - 无需复杂的连接池管理

| 组件 | 规范 |
|------|------|
| **操作系统** | Linux (Ubuntu 22.04+/Debian 12+) |
| **容器运行时** | Containerd / CRI-O |
| **编排平台** | Kubernetes 1.25+ |

| **网络插件** | CNI兼容 (Calico/Flannel/Cilium) |
| **存储** | CSI兼容卷 (本地临时卷或NFS) |

### 5.3 资源限制建议

| 容器 | 内存请求 | 内存限制 | CPU请求 | CPU限制 |
|------|----------|----------|---------|---------|
| MCP Server | 128Mi | 256Mi | 100m | 500m |
| Browser Container | 1Gi | 2Gi | 500m | 2000m |
| **注意**：根据实际使用情况调整，特别是内存密集型SPA |

### 5.4 安全配置

| 安全项 | 配置 |
|--------|------|
| **容器安全中心** | 以非root用户运行 (UID 10001) |
| **能力 dropping** | 删除所有Linux能力，仅添加必要项 |
| **文件系统** | 只读根文件系统，仅挂载必要可写目录 |
| **网络策略** | 仅允许Pod内通信和必要出站流量 |
| **秘密管理** | 通过K8s Secret注入加密密钥 |
| **镜像安全** | 使用内部镜像仓库，启用漏洞扫描 |
| **运行时安全** | 启用Falco或similar进行异常行为检测 |

### 5.5 监控和日志

| 监控项 | 实现方式 |
|--------|----------|
| **应用指标** | 通过Prometheus客户端库暴露 |
| **业务指标** | 登录成功率、平均响应时间、会话命中率 |
| **系统指标** | 内存使用率、CPU使用率、GC频率 |
| **日志结构** | JSON格式，包含trace ID和上下文 |
| **健康检查** | livenessprobe (存活) + readinessprobe (就绪) |
| **告警阈值** | 内存>85%、CPU>90%、错误率>5%（5分钟窗口） |

## 6. 验收标准

### 6.1 功能验收标准

| 功能 | 验收条件 |
|------|----------|
| **导航和基本交互** | 能够打开任意URL，执行点击、输入、选择等基本操作 |
| **会话持久化** | 能够保存登录状态并在后续会话中恢复，跨Pod共享 |
| **智能等待** | 自动处理SPA hydration和异步加载，无需手动sleep |
| **错误处理** | 对常见错误（元素不存在、超时、网络问题）提供清晰错误信息 |
| **资源保护** | 在内存/CPU接近限制时自动降级或拒绝新请求 |
| **OpenClaw集成** | 通过标准配置文件即可完成集成，无需复杂设置 |
| **安全合规** | 不泄露敏感信息，敏感数据在传输和存储中加密 |
| **可观测性** | 提供足够的日志和指标用于故障排除和性能分析 |

### 6.2 性能验收标准

| 指标 | 验收标准 |
|------|----------|
| **冷启动时间** | Pod就绪时间 < 30秒（从kubectl apply开始） |
| **导航延迟** | 从navigate命令到页面加载完成 < 5秒（普通页面） |
| **操作延迟** | 单个元素操作（click/type） < 500ms（网络稳定时） |
| **会话存储延迟** | 会话保存/恢复 < 50ms |
| **并发处理** | 能够处理 20 个并发登录测试而不显著降级 |
| **内存使用率** | 稳态下内存使用率 < 70%（有足够余量应对突发） |
| **CPU使用率** | 稳态下CPU使用率 < 60%（有足够余量应对突发） |
| **垃圾回收** | GC暂停时间 < 50ms，频率 < 每10秒一次 |

### 6.3 可靠性验收标准

| 指标 | 验收标准 |
|------|----------|
| **Pod重启率** | 正常运行下每天 < 0.1次（因应用崩溃导致） |
| **浏览器崩溃恢复** | 自动恢复率 > 99%（通过健康检查重启容器） |
| **会话一致性** | 100%的会话保存和恢复操作成功 |
| **故障隔离** | 单个Pod故障不影响其他Pod的正常运行 |
| **升级回滚** | 支持无缝滚动更新和快速回滚 |
| **备份恢复** | 会话数据支持定期备份和从备份恢复 |
| **灾难恢复** | 从完备副本恢复时间 < 15分钟 |

### 6.4 安全验收标准

| 指标 | 验收标准 |
|------|----------|
| **敏感数据泄露** | 日志、错误信息、内存中不含明文密码或token |
| **加密强度** | 会话数据使用AES-256-GCM加密 |
| **密钥管理** | 加密密钥通过K8s Secret管理，不在镜像或代码中 |
| **容器特权** | 以非root用户运行，最小必要能力集 |
| **网络暴露** | 仅暴露必要端口（如3001供OpenClaw调用），其他默认封闭 |
| **依赖安全** | 所有npm依赖无已知高危漏洞（依赖审计通过） |
| **镜像安全** | 基础镜像无高危漏洞，使用distroless或similar基础镜像 |
| **日志安全** | 日志中不含敏感信息（自动脱敏） |
| **合规审计** | 通过基本的GDPR/CCPA合规检查（数据最小化、目的限制） |

## 7. 风险与对策

### 7.1 已识别风险及应对措施

| 风险 | 可能性 | 影响 | 应对措施 |
|------|--------|------|----------|

| **共享Volume性能瓶颈** | 低 | 中 | 1. 使用高性能存储类（SSD-based）<br>2. 会话数据采用增量存储<br>3. 考虑使用tmpfs for临时会话 |

| **会话存储空间耗尽** | 中 | 中 | 1. 实现会话过期清理机制<br>2. 设置会话大小警告阈值<br>3. 提供会话存储监控和告警 |
| **跨节点会话共享需求** | 低 | 高 | 1. 明确说明当前方案仅支持同节点会话共享<br>2. 如需跨节点，建议使用分布式文件系统（如NFS、CephFS）<br>3. 提供迁移路径指南 |
| **浏览器版本兼容性** | 低 | 中 | 1. 锁定具体的Chromium版本<br>2. 提供版本升级指南和回滚过程<br>3. 监控Playwright官方兼容性报告 |
| **资源争用导致Pod驱逐** | 中 | 高 | 1. 设置合理的资源请求和限制<br>2. 实现Pod级别的资源监控和告警<br>3. 使用Pod优先级和抢占策略 |
| **网络分区导致的故障** | 低 | 高 | 1. 依赖Kubernetes的网络容错能力<br>2. 实现应用级别的重试机制<br>3. 监控网络健康状态 |
